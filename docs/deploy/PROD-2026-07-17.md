# Déploiement PROD — 2026-07-17

Trace du déploiement de la couche WSP + du Passport public sur la production Opus X.
Document de traçabilité (non normatif) : il consigne ce qui a été fait, dans quel ordre,
et ce qui reste dû.

## Contexte

- **Cible** : production Opus X, projet Supabase ref **`lhqephwy`** (masqué au-delà des
  8 premiers caractères).
- **Particularité** : la prod a été **montée à la main** (dashboard), elle ne possède
  **aucune table `supabase_migrations.schema_migrations`**. Il n'existe donc aucun
  historique CLI.
- **Conséquence sur la méthode** : déploiement **manuel via le SQL Editor** du dashboard,
  migration par migration. Un `supabase db push` aurait cru l'historique vide et **rejoué
  les migrations 1→5 sur des tables déjà peuplées** (2 profils / 2 passports), avec le
  risque d'abort (CHECK de la migration 5) et de contournement du schéma existant. Le
  déploiement manuel était donc le seul chemin sûr.

## Exclues volontairement

- **11 — `20260713000006_wsp_issuer_registration.sql`** : le seed pose un Issuer
  `status='active'` avec `redirect_uri = http://localhost:3000/…` et `hmac_key_ref = null`.
  Deux blocages : le `redirect_uri` localhost casse l'égalité exacte exigée par
  `wsp_authorize_issuer`, et le trigger `wsp_issuers_no_mutation` **interdit tout UPDATE**
  (service_role compris) — un Issuer mal enregistré serait **irréparable**. L'Issuer réel
  sera enregistré **plus tard**, comme acte d'exploitation, avec sa **vraie URL WTS de prod**
  et son **secret HMAC** (posé hors-repo).
- **14 — `20260717000003_public_passport_demo_showcase.sql`** (Lot 5, `is_demo` + vitrine
  démo) : **non committée**, mise de côté. Le code applicatif correspondant l'est aussi.

- `profiles.opus_id` **UNIQUE** (charnière : les FK des tables WSP référencent `profiles(opus_id)`).
- Les **5 fonctions Sprint-1** présentes (`generate_ulid`, `generate_opus_id`,
  `generate_unique_handle`, `slugify`, `touch_updated_at`).
- Schéma **`extensions`** présent (requis par `create extension pgcrypto`).
- `consents.version = v1.0.0` (conforme au CHECK `^v[0-9]+\.[0-9]+\.[0-9]+$`) → **R3 neutralisé**.

## Risques traités au palier 0

- **R1** — Issuer avec `redirect_uri` localhost figé par le trigger `no_mutation` :
  **traité par exclusion** de la migration 11.
- **R2** — ordre d'application : la migration 12 **ferme le Point B** (`revoke select on
  passports from anon`), qui annule le `grant` de la migration 3. Vérifié après application :
  `anon_peut_lire_passports = false`, vue à 4 colonnes sans `profile_id`.
- **R3** — CHECK de format sur `consents.version` : **neutralisé par constat prod**
  (`consents.version` déjà en `v1.0.0`).
- **R4** — `drop table if exists public.wsp_fact_revocations cascade` (rejeu destructeur d'un
  magasin append-only) : **neutralisé** (fusionné en `create table if not exists`),
  **committé `f0c3234`**.

## Dette ouverte

- **Pas de `schema_migrations` en prod.** Si l'on branche le CLI Supabase un jour, il faudra
  d'abord **marquer les migrations 1→13 comme déjà appliquées** (baseline / `migration repair`)
  **avant tout `db push`** — sinon le CLI les rejouerait, avec rejeu destructeur possible.
- L'Issuer réel (ex-migration 11) reste à enregistrer avec sa vraie URL + secret.
- Le Lot 5 (`is_demo` + vitrine démo) reste non déployé, sur `web/web-003-registry`.
